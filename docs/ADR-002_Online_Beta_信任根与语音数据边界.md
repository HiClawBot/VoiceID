# ADR-002：Online Beta 信任根与语音数据边界

状态：Accepted for staged implementation
日期：2026-07-21
适用范围：VoiceID Online Beta v0.2 及后续受控 Voice Pilot

## 1. 背景

Browser Playground v0.1 的 `VoiceProof`、钱包签名结果和 `AuthSession` 均在客户端生成。它们适合验证交互与对象编排，但浏览器、DOM、前端分数和 `sessionStorage` 都可被修改，不能成为线上认证的信任根。

同时，声音可能属于敏感个人信息或生物识别信息。浏览器无法可靠证明音频来自物理麦克风，也无法完全阻止回放、合成语音或虚拟音频设备注入。系统必须先确定强认证、会话签发、语音用途和数据生命周期，再实现真实模型。

## 2. 决策

### 2.1 强认证是会话信任根

- 普通用户使用 Passkey；钱包用户可使用服务端验证的 SIWE。
- 所有 challenge 与 nonce 由服务器签发，具有短期过期时间，并在事务中只消费一次。
- Passkey 验证覆盖 RP ID、origin、challenge、UP/UV 与凭证状态。
- SIWE 验证覆盖消息严格解析、domain、URI、address、chain ID、nonce、issued-at、expiration-time，以及 EOA/ERC-1271 路径。
- 只有服务端验证通过后才能签发认证会话。客户端字段或客户端签名返回本身不能把 `serverVerified` 升级为 `true`。

### 2.2 会话由服务器控制

- 使用随机、不可预测的服务器会话标识，浏览器仅持有 `Secure; HttpOnly; SameSite` Cookie。
- 登录后轮换会话标识，并同时执行 idle expiry 与 absolute expiry。
- 退出、凭证撤销、账户停用、风险事件与管理员处置都可立即撤销会话。
- 审计记录只保留必要事件、结果、风险原因与关联 ID，不记录原始音频、完整签名消息、签名值、PIN 或声纹模板。

### 2.3 语音默认是 observe-only 策略信号

- `VoiceAssessment` 分别表达音频质量、语音活动、挑战短语匹配、说话人相似度和 PAD/回放风险，不再使用单一 `verified` 状态掩盖不同结论。
- Online Beta 初始阶段只记录去标识质量指标和实验结果；语音失败不阻止用户通过 Passkey 或 SIWE 建立、使用或恢复账户。
- 声音不得成为唯一登录、唯一恢复或高价值操作批准因子。
- 只有模型评估、攻击测试、隐私影响评估和独立安全审查全部过门后，Voice Pilot 才能批准有限门控用途。

### 2.4 服务端短暂处理音频是有条件决策

为了让短语匹配、说话人模型与 PAD 策略在服务器侧可复现并不可被客户端伪造，受控实验可以在隔离推理服务内短暂处理音频，但必须先满足以下条件：

- 完成单独同意、撤回同意、替代认证路径和删除账户流程。
- 批准 PIPIA/DPIA、适用地区、年龄边界、数据区域和处理者清单。
- 音频经加密通道进入不暴露公网的推理服务，只存在于内存或 `tmpfs`。
- 成功、失败、取消、超时和进程异常路径都验证删除；原始音频不得进入数据库、对象存储、日志、错误追踪或备份。
- 声纹模板使用每用户数据密钥加密，密钥由 KMS 包装；数据库不保存明文 embedding。

在上述条件未全部满足前，不接收真实用户音频到服务器，不建立生产声纹模板。

### 2.5 Playground 与 Online Beta 隔离

- `/beta/` 保持 Browser Playground v0.1，继续使用 `demo-only`、模拟钱包和 `serverVerified=false`。
- Online Beta 使用独立应用与服务端契约；生产构建不得包含模拟音频、模拟钱包、演示 PIN 或演示恢复密钥路由。
- Playground 对象不能直接迁移为生产凭证或 enrollment 数据。
- WebAuthn RP ID、SIWE domain、Cookie domain、TLS origin、邀请地区和数据区域必须在 Sprint 1 开工前冻结。

## 3. 被否决方案

### 方案 A：相信客户端模型分数并直接签发会话

否决。前端分数和控制流可被篡改，服务器无法证明模型、输入或策略真实执行。

### 方案 B：把声音作为唯一认证或恢复因子

否决。回放、合成、虚拟输入、环境差异、误拒和不可更换性使其不适合作为唯一因子。

### 方案 C：为了隐私只做浏览器本地推理，同时把结果用于服务端放行

否决用于认证。该方案可用于 UX 与 Playground，但服务端不能信任客户端返回的结论。

### 方案 D：长期保留原始音频以便调参和客服复核

否决。它显著扩大敏感数据暴露、同意用途、删除、备份和内部访问风险。研究数据集必须另立目的、数据源、同意与治理流程。

## 4. 后果

正面影响：

- 强认证、语音实验和产品愿景各自有清晰边界。
- 可以先交付真实 Passkey/SIWE 会话，不被声纹模型研究阻塞。
- 语音策略具备可追溯的分阶段结果、模型版本和停用开关。
- 删除、撤回、审计与回滚可在接触真实生物数据前验证。

成本与限制：

- 需要 Auth API、数据库、KMS、隔离推理服务和三环境，静态 GitHub Pages 不再足够承载 Online Beta。
- 服务端短暂处理音频需要更严格的隐私、法律、安全和平台责任。
- 浏览器仍不能可靠证明物理音频来源，因此 Voice Pilot 也不能消除回放与注入风险。

## 5. 验证与回滚

- 自动化验证 Passkey/SIWE challenge 单次消费、过期、拒绝、重放、撤销和并发路径。
- 用攻击集评估错误短语、录音回放、TTS、虚拟输入、噪声与设备差异，并按人群与环境切片。
- 对成功、失败、取消、超时和崩溃路径执行原始音频删除证明。
- 通过 feature flag 可独立关闭 voice enrollment 与 verification，不影响 Passkey/SIWE 登录。
- 任何 Critical/High 安全问题、原始音频残留、删除任务失败或未经批准的数据区域漂移都会触发 No-Go 或回滚。

## 6. 非目标

本 ADR 不批准主网交易、资产授权、私钥/助记词托管、KYC、唯一人类证明、DID Registry、ZK nullifier、第三方生产 SDK 或用声音批准高价值操作。
